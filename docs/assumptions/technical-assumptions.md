# Technical Assumptions

These are the assumptions baked into the current design.
If any assumption is invalidated, update this document and review the affected components.

---

## Runtime & Platform

| Assumption | Impact if wrong |
|-----------|----------------|
| Node.js 22 LTS is the runtime | OTel SDK and Prisma versions may need updating for older Node versions |
| Linux containers (Alpine-based) for production | `bcrypt` native bindings may behave differently on Windows dev machines — use Docker for parity |
| Single-region deployment for v1.0 | Multi-region adds clock drift concerns for JWT expiry and Redis replication complexity |

## Database

| Assumption | Impact if wrong |
|-----------|----------------|
| PostgreSQL 16 is the database | Prisma adapter swap needed for MySQL/SQLite; some PG-specific behaviours (e.g. `uuid` type) differ |
| `@prisma/adapter-pg` native driver is used | The Prisma URL connection (without adapter) has different connection pooling behaviour |
| All UUIDs generated by PostgreSQL `uuid()` default | Changing to application-generated UUIDs requires schema migration |
| Soft-deleted records are never queried without the `deletedAt: null` filter | Missing filter silently returns deleted records — all BaseRepository methods enforce this |
| A single database schema is used (no multi-tenancy) | Adding per-tenant schemas requires major refactoring of PrismaService and all queries |

## Authentication

| Assumption | Impact if wrong |
|-----------|----------------|
| JWT access token lifetime of 15 minutes is acceptable | Shorter lifetimes reduce risk but require more frequent refresh calls |
| Refresh token rotation (single-use) is the primary replay-attack mitigation | Concurrent refresh requests (race condition) can cause token invalidation — acceptable for v1.0 |
| API keys are stored as bcrypt hashes with the prefix stored plaintext | The prefix must be globally unique enough to look up keys without full-scan; current format `<6-char-random>_<hash>` satisfies this |
| Email uniqueness is enforced at DB level | If that constraint is removed, duplicate accounts will appear and auth will be ambiguous |

## Infrastructure

| Assumption | Impact if wrong |
|-----------|----------------|
| Redis is always available (no graceful BullMQ degradation) | If Redis is down, queue operations throw `SRV0002`; app continues to serve non-queue requests |
| OTel Collector is optional (`OTEL_ENABLED=false` by default) | Disabling OTel removes distributed tracing; logs still work via Pino |
| All app instances share the same Redis and JWT secrets | Horizontal scaling without shared Redis breaks BullMQ; mismatched JWT secrets cause 401s |
| Docker Compose is sufficient for local dev | Teams using Kubernetes locally (minikube/kind) will need to adapt compose to helm |

## Security

| Assumption | Impact if wrong |
|-----------|----------------|
| HTTPS termination happens at the load balancer/proxy, not in the app | Running the app directly on HTTPS requires adding TLS config to `main.ts` |
| Rate limiting via `ThrottlerModule` (in-memory by default) is acceptable for v1.0 | In-memory rate limiting does not work correctly when multiple app instances run; switch to Redis-backed throttler |
| `bcrypt` with 12 rounds is secure enough for password hashing | If cracking tools improve significantly, increase `BCRYPT_ROUNDS` (each +1 doubles hash time) |

## Testing

| Assumption | Impact if wrong |
|-----------|----------------|
| E2E tests run against a real PostgreSQL test database | Using SQLite in-memory for tests would miss PG-specific constraint behaviour |
| 80% line coverage is the minimum acceptable threshold | Lower coverage may leave critical error paths untested |
